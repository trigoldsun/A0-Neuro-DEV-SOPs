# 微服务环境部署与运维最佳实践手册

> **版本**: v1.1.0
> **更新时间**: 2026-05-12
> **说明**: 最佳实践跨环境通用（Dockerfile、Compose、迁移等），但部署位置因环境而异：
> - **Dev/Test** → 开发者本地 Docker（独立 `-p` 项目）
> - **UAT** → 阿里云 ECS（`platformctl` 管理）

## 1. 引言

本手册旨在为基于《系统开发和文档管理指导手册 v2.2.0》[1]（以下简称“《一号文件》”）构建的微服务系统，提供一套使用Docker进行部署和运维的最佳实践。手册涵盖 Dockerfile 编写、Docker Compose 编排、数据库迁移、密钥管理、监控日志等方面。这些实践在 Dev/Test（本地宿主机）和 UAT（阿里云ECS）环境中通用。

## 2. Dockerfile 编写最佳实践

高质量的Dockerfile是构建高效、安全、可维护Docker镜像的基础。遵循以下实践，可以优化镜像大小、构建速度和运行时性能。

### 2.1 多阶段构建 (Multi-stage Builds)

利用多阶段构建可以显著减小最终镜像的体积，只包含运行时所需的最小组件，从而提高部署速度和安全性。

```dockerfile
# 阶段1: 构建阶段
FROM golang:1.22-alpine AS builder
WORKDIR /app
COPY go.mod go.sum ./ # 复制依赖文件
RUN go mod download   # 下载依赖
COPY . .
RUN CGO_ENABLED=0 GOOS=linux go build -a -installsuffix cgo -o main . # 编译Go应用

# 阶段2: 运行阶段
FROM alpine:latest # 使用极小的alpine作为基础镜像
WORKDIR /app
COPY --from=builder /app/main . # 从构建阶段复制编译好的二进制文件
# 如果有静态资源或配置文件，也在此阶段复制
# COPY --from=builder /app/configs ./configs

EXPOSE 30001 # 暴露服务监听端口，符合《一号文件》端口规范
USER nobody # 以非root用户运行，提高安全性
ENTRYPOINT ["./main"]
```

### 2.2 使用轻量级基础镜像

选择官方提供的最小化基础镜像，如 `alpine` 或 `scratch`，可以有效减少镜像层数和最终镜像大小。

### 2.3 以非 Root 用户运行

在Dockerfile中使用 `USER` 指令，指定容器运行时使用非 Root 用户，可以有效降低容器被攻破后的权限风险。

### 2.4 明确暴露端口

使用 `EXPOSE` 指令声明容器将监听的端口，这有助于文档化和网络配置。确保端口符合《一号文件》中30000-50000的端口段要求。

### 2.5 添加健康检查

使用 `HEALTHCHECK` 指令定义容器的健康检查机制，帮助Docker和编排工具判断服务是否正常运行。

```dockerfile
HEALTHCHECK --interval=30s --timeout=3s --start-period=5s --retries=3 CMD curl -f http://localhost:30001/health || exit 1
```

## 3. Docker Compose 最佳实践

Docker Compose是管理多容器微服务应用的利器，在 Dev/Test 和 UAT 环境中通用。

### 环境差异化策略

| 方面 | Dev/Test（本地） | UAT（ECS） |
|:---|:---|:---|
| Compose 文件 | 项目级 `docker-compose.yml`（仅含项目服务） | 平台 `docker-compose.infra.yml` + 项目 `docker-compose.yml` |
| 项目隔离 | `docker compose -p <项目编号>` | 无需额外参数 |
| 基础设施 | 连接 UAT 共享服务或本地副本 | 平台 Tier1（PG/Redis/MinIO/Nginx） |
| 端口映射 | localhost:3xxxx | ECS 公网IP:3xxxx |

### docker-compose.yml 模板

### 3.1 统一网络管理

为所有服务定义一个共享的自定义网络，使服务之间可以通过服务名互相通信，而无需关心IP地址。

```yaml
networks:
  app_network:
    driver: bridge
```

### 3.2 资源限制与配额

严格按照《一号文件》中“单服务CPU限制在4核及以下，内存8G及以下”的要求，为每个服务配置资源限制，防止资源滥用。

```yaml
services:
  your_service:
    # ...
    deploy:
      resources:
        limits:
          cpus: '0.5' # 限制CPU为0.5核
          memory: 512M # 限制内存为512MB
```

### 3.3 数据持久化

对于数据库、缓存、对象存储等需要持久化数据的服务，务必使用Docker Volume进行数据存储，确保容器重启或更新时数据不丢失。

```yaml
volumes:
  db_data:
  redis_data:
  minio_data:

services:
  db:
    # ...
    volumes:
      - db_data:/var/lib/postgresql/data
```

### 3.4 环境变量与敏感信息管理

*   **`.env` 文件**: 使用 `.env` 文件管理非敏感的环境变量，并在 `docker-compose.yml` 中通过 `env_file` 引用。
*   **密钥管理**: 敏感信息（如数据库密码、API密钥）**禁止明文写入代码或配置文件**。应通过密钥管理服务（如Vault Agent）在容器启动时注入，或通过Docker Compose的 `secrets` 功能（虽然在单机非生产环境可能略显复杂，但可作为未来向生产环境过渡的考虑）。

### 3.5 服务依赖与启动顺序

使用 `depends_on` 确保服务按正确的顺序启动，例如数据库服务应在业务服务之前启动。

```yaml
services:
  backend:
    depends_on:
      - db
      - redis
```

### 3.6 统一日志与监控配置

将日志收集器和监控系统作为共享服务部署。在 UAT 环境中，VictoriaMetrics 和 Vector 已作为 Tier1 基础设施运行。

## 4. 数据库版本控制最佳实践

《一号文件》强制要求使用代码化的数据库迁移工具。在Docker环境中，这可以通过以下方式实现：

### 4.1 独立的迁移容器

创建一个独立的Docker服务，专门用于执行数据库迁移。该服务应在业务服务启动前运行，确保数据库结构是最新的。

```yaml
services:
  db_migrate:
    build:
      context: ./db_migrations # 数据库迁移工具的Dockerfile目录
      dockerfile: Dockerfile
    container_name: db_migration_tool
    environment:
      DATABASE_URL: "postgres://user:password@db:5432/mydb?sslmode=disable"
    depends_on:
      - db # 确保数据库服务已启动
    command: ["./migrate", "up"] # 示例：执行迁移命令
    restart: "no" # 迁移成功后无需重启
```

### 4.2 迁移脚本与工具

使用如Goose或golang-migrate等工具，将数据库结构变更定义为版本化的脚本。这些脚本应与应用代码一同进行版本控制。

## 5. 密钥管理最佳实践

遵循《一号文件》中“敏感配置禁止明文写入代码或配置文件”的要求，采用安全的密钥管理方案。

### 5.1 环境变量文件管理

**Dev/Test（本地）**：使用项目根目录的 `.env` 文件管理配置，**确保 `.env` 已纳入 `.gitignore`**，严禁提交到 Git。

**UAT（ECS）**：敏感凭证存储在 `/opt/neuro-devsituat-platform/.env.secrets`（权限 600），通过环境变量注入容器。

### 5.2 环境变量注入

应用容器通过环境变量获取敏感信息。这些环境变量不应直接写入 `docker-compose.yml`，而应由Vault Agent或其他安全机制在容器启动时动态设置。

### 5.3 挂载敏感文件

对于需要文件形式的敏感信息（如证书），Vault Agent可以将其写入临时文件，并通过Docker Volume挂载到应用容器中。

## 6. 监控与日志最佳实践

建立完善的监控和日志系统，是微服务运维的关键。

### 6.1 统一日志收集

使用Vector作为统一的日志收集器，收集所有容器的日志以及宿主机日志，并将其发送到集中的日志存储系统（如Loki、VictoriaMetrics Logs或Elasticsearch）。

### 6.2 指标监控

使用VictoriaMetrics作为指标存储，各微服务通过OpenTelemetry或Prometheus客户端暴露指标，VictoriaMetrics负责抓取和存储。配合Grafana进行可视化。

### 6.3 链路追踪

采用OpenTelemetry纯埋点的方式，在Go/Gin应用中集成链路追踪。通过TraceID将日志、指标和调用链关联起来，便于故障排查。

## 7. 部署与更新策略

### 7.1 一键部署

通过 `docker compose up -d` 命令实现所有服务的快速部署。确保 `docker-compose.yml` 文件定义清晰、完整。

### 7.2 持续集成/持续部署 (CI/CD)

集成CI/CD流程，自动化代码构建、镜像构建、测试和部署。例如，当代码提交到特定分支时，自动触发Jenkins/GitLab CI/GitHub Actions等工具进行构建和部署。

### 7.3 滚动更新与回滚

在非生产环境，虽然通常是单实例部署，但仍应规划好更新流程。更新服务时，先构建新镜像，然后使用 `docker compose up -d --build <service_name>` 进行更新。若出现问题，应能快速回滚到上一个稳定版本。

### 7.4 环境清理

#### Dev/Test（本地）

定期清理本地 Docker 构建缓存和未使用的镜像，释放磁盘空间：

```bash
# 清理所有未使用的镜像、容器、网络和构建缓存
docker system prune -a -f

# 清理特定项目的已停止容器
docker compose -p <项目编号> down -v
```

#### UAT（ECS）

```bash
# 由 platformctl 统一管理
platformctl resources   # 查看磁盘占用
df -h                   # 确认磁盘使用率
docker system prune -a -f
```
