# 01-系统开发和文档管理指导手册（简称：一号文件）

> **版本**: v2.3.0
> **更新时间**: 2026-05-12
> **更新说明**: 补充非生产环境架构定义：Dev/Test（本地Docker独立项目）与 UAT（阿里云ECS共享平台）；新增本地开发环境隔离规范。

## 1. 开发与架构规范（L1级强制执行）

### 1.1 核心开发铁则

在系统开发过程中，必须严格遵守以下核心铁则：

1. **架构分层**：禁止逆向调用，依赖层级**不得超过3层**。
2. **资源分配**：IP、端口与计算资源需**先登记后使用**。单服务CPU限制在**4核及以下**，内存**8G及以下**。
3. **端口规范**：统一使用**30000-50000段**。非整百端口分配给开发与测试环境，整百端口分配给生产前端。
4. **安全基线**：强制要求**参数化查询**防SQL注入、**输出转义**防XSS，且接口必须携带**幂等键**。
5. **接口规范**：采用**RESTful风格**，路径带版本号，统一返回格式，并必填**限流响应头**。
6. **API废弃策略**：API旧版本至少保留**6个月兼容期**，下线前30天发布废弃通知，并在响应头中标注 `Deprecated`。
7. **质量门禁**：单元测试覆盖率必须达标，PR需**至少1人审批**，安全扫描必须**零高危漏洞**。
8. **开发流程**：任务拆分至**30分钟内可完成**粒度；严格执行**TDD（测试驱动开发）**红黄绿流程；在**独立工作树分支**开发，完成后销毁。
9. **调试与上线**：调试必须**先复现根因**，上线必须提供**可量化验证证据**。
10. **命名规范**：产品侧**统一前缀并隔离内部代号**；开发侧采用**自定义前缀加数字**（如w001）并在内部统一使用。

### 1.2 技术栈与选型策略

技术选型需兼顾开发测试环境的**极致降本**与生产环境的**合规可靠**，同时保证**代码与协议的一致性**以实现平稳过渡。

#### 1.2.1 核心语言与展现层

1. **后端主语言**：强制采用 **Go/Gin**。利用其极低内存占用与高并发特性，单台低配裸机即可承载数十个微服务实例。
2. **前端框架**：统一采用 **Vue 3 结合 UniApp**。实现一套代码多端发布，极限降低适配工作量。
3. **备选审批项（Java）**：在开发**企业级复杂系统**或**事务密集型服务**时，允许将 **Java（JDK 17/21 LTS）**作为备选架构。但必须提前提交 ADR 文档说明选型理由，经**架构委员会审批通过**后方可使用。

#### 1.2.2 流量与基础设施层

1. **Dev/Test（本地宿主机）**：开发者本地 Docker，每个项目独立 Docker Compose 项目（使用 `-p <项目编号>` 参数），容器、网络、数据卷完全隔离，无共享基础设施。
2. **UAT（阿里云ECS）**：单节点 Nginx、PostgreSQL 18、Redis 7、MinIO、VictoriaMetrics 作为共享 Tier1 基础设施，通过 platformctl 统一管理。
3. **生产环境**：网关必须采用**云厂商 SLB/ALB** 扛 DDoS 攻击并保障入口高可用；内部微服务网关使用 **APISIX 集群**；配置中心与服务发现采用 **Nacos 或 Consul 的高可用集群**。
4. **平稳过渡**：由于底层协议一致，代码层面**对接标准 API**，无需修改即可实现环境切换。

#### 1.2.3 数据与存储层

1. **Dev/Test（本地宿主机）**：可连接 UAT 共享基础设施（PostgreSQL 18、Redis 7、MinIO），或本地运行轻量副本；每个项目独立，资源不共享。
2. **UAT（阿里云ECS）**：共享**单机自建 PostgreSQL 18**、**单机自建 Redis 7**、**单机自建 MinIO**，资源通过资源登记表（附录A-F）统一管理，防止冲突。
3. **生产环境**：数据丢失或宕机是灾难性的，必须购买云厂商的 **RDS PostgreSQL（主从高可用）**、**云托管 Redis（集群版）**以及**云厂商 OSS**，以获取自动备份、容灾、高可用和合规审计能力。
4. **版本控制**：全环境强制使用代码化的数据库迁移工具（如 **Goose** 或 **golang-migrate**），**禁止手动直连改库**。
5. **平稳过渡**：自建版与云托管版在协议和驱动上完全一致（MinIO 兼容 S3），切换环境时代码层面的 **SDK 零修改**。

#### 1.2.4 核心中间件层

1. **开发测试环境**：使用 **Redis Streams** 处理轻量消息，并使用 **Asynq** 进行任务调度。
2. **生产环境**：涉及财务对账、核心订单等必须保证消息零丢失的场景，必须上**云托管 Kafka 或 RocketMQ**；非核心业务可继续使用 Redis Streams；任务调度全环境保持一致使用 Asynq。
3. **平稳过渡**：若生产环境有高吞吐要求，需在代码层面对消息队列进行**接口抽象**，开发环境注入 Redis 实现，生产环境注入 Kafka 实现。

#### 1.2.5 安全与风控层

安全与风控层在**全环境必须保持一致**，必须在开发测试阶段就严格跑通：

1. **认证与阻断**：采用 **JWT 结合 Redis 黑名单**满足实时阻断需求。
2. **设备风控**：使用 **FingerprintJS 结合自研引擎**替代收费商业版。
3. **加密算法**：强制采用**国密算法（SM2/SM3/SM4）**满足金融合规底线。
4. **密钥管理**：敏感配置（如 AK/SK、数据库密码）**禁止明文写入代码或配置文件**，必须通过密钥管理服务（如 Vault 或云厂商 KMS）注入，且审计所有访问记录。

#### 1.2.6 第三方服务与触达层

第三方服务与触达层同样在**全环境保持一致**：

1. **消息触达**：常规通知走**微信体系**（公众号/小程序/模板消息）及 **App 内推**免费通道，仅合规强制环节用短信。
2. **实名认证**：采用**银企直连与人脸核身**，并强制执行**延迟认证策略**，杜绝羊毛党刷爆接口费。

#### 1.2.7 监控与运维层

1. **开发测试环境**：采用**单机 VictoriaMetrics**、**Vector** 以及 **OpenTelemetry（纯埋点打入日志）**。
2. **生产环境**：采用 **VictoriaMetrics 集群**、**Vector**、**OpenTelemetry 纯埋点**以及**便宜的冷存储**。
3. **降本策略**：放弃笨重的 ELK 和 Jaeger 服务端，采用 VictoriaMetrics 配合纯埋点 TraceID 关联日志的轻量方案，足以满足生产环境绝大部分的故障排查需求，同时**省下海量服务器成本**。

## 2. 环境与部署规范

1. **生产环境**：必须严格执行高可用要求，中间件副本数需**大于等于3**，服务必须**多实例部署**。
2. **非生产环境**：研发、测试、UAT 等环境允许**单副本、单实例部署**以节省资源。
3. **环境隔离**：所有测试环境必须**明确标识**，严禁接入生产流量或存储生产敏感数据。
4. **容灾与备份**：生产数据库**每日自动备份**，RPO≤1小时，RTO≤30分钟，**每季度至少执行一次**容灾恢复演练。
5. **灰度与回滚**：所有生产发布必须支持**灰度发布**（如先放量5%→20%→100%），并预置**一键回滚能力**，回滚时间不超过5分钟。

## 3. 团队协作与文档管理

### 3.1 团队协作规范

1. **代码评审**：PR 提交后，评审人需在 **1个工作日内** 响应，禁止积压。
2. **技术债务**：所有技术债务必须在 Issue 跟踪系统中登记，并安排在后续迭代中逐步消化。
3. **架构决策**：所有重大架构变更必须提交 ADR（架构决策记录）文档，经架构委员会审批后归档。

### 3.2 目录与归档

1. **标准目录**：项目目录统一划分为 `00-docs`（技术/规范）、`01-dev-project`、`02-subagent-project`、`03-uiux-project` 及 `project_records`（全量归档）。
2. **留存要求**：所有项目产出（包括会议纪要、版本文档、代码提交记录、评审记录等）必须**永久留存**，且修改记录必须**可追溯**。
3. **归档原则**：归档文件**仅可新增，禁止删除或篡改**。

### 3.3 命名与版本

1. **文件命名**：必须**见名知意**，并带版本号后缀（如 `文件名v1.0.0.md`）。
2. **过时文档**：需重命名为 `过时-原文件名.md.bak.YYYYMMDD` 并归档。
3. **版本控制**：遵循**语义化版本**（v主版本.次版本.补丁号），重大升级必须提供**迁移文档**。

## 附录：参考指南

### A. 文档分类级别

| 级别 | 分类 | 用途 |
| :--- | :--- | :--- |
| L1 | policy | 核心制度，最高权威 |
| L2 | regulation | 强制规定，违反有后果 |
| L3 | guideline | 指导性建议 |
| L4 | standard | 公开/内部标准 |
| L5 | instruction | 操作手册/接口规范 |
| L6 | reference | 参考资料 |

### B. 常用排查命令

1. **端口占用**：`lsof -i :端口号`
2. **编码检查**：`file -i 文件名`
3. **快速归档**：`mv 原文件名 过时-原文件名.md.bak.$(date +%Y%m%d)`
